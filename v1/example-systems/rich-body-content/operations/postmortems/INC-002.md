# INC-002

## SUMMARY

The authentication service entered a crash loop at 09:15 UTC on 2026-03-02 due to an expired internal TLS certificate. The failure cascaded to all downstream services that required auth token validation, causing a platform-wide outage lasting 1 hour 23 minutes.

## TIMELINE

### Detection

- 09:15 — Auth service pods begin crash-looping
- 09:17 — PagerDuty fires: auth-service health check failing across all replicas
- 09:18 — On-call engineer acknowledges

> Alert: "auth-service 0/3 replicas ready, last restart reason: CrashLoopBackOff"

### Investigation

- 09:20 — Checked pod logs: TLS handshake failure connecting to the internal token signing service
- 09:24 — Confirmed the internal CA certificate expired at 09:00 UTC
- 09:26 — Checked certificate management: the auto-renewal cron had been disabled during a maintenance window on 2026-02-15 and never re-enabled

> Pod log: "x509: certificate has expired or is not yet valid: current time 2026-03-02T09:15:00Z is after 2026-03-02T09:00:00Z"

### Cascade

- 09:22 — API gateway begins rejecting all requests: cannot validate tokens
- 09:25 — Dashboard, mobile app, and webhook delivery all report errors
- 09:28 — Background job workers pause: they validate service tokens before processing

### Mitigation

- 09:30 — Generated new internal CA certificate manually
- 09:42 — Distributed certificate to all services via config update
- 09:48 — Auth service pods recover, begin accepting connections
- 09:55 — API gateway resumes normal traffic
- 10:05 — Background job backlog fully drained
- 10:38 — All services confirmed healthy, incident closed

## ROOT_CAUSE

The internal certificate authority uses a cron job to rotate certificates 30 days before expiry. During a planned maintenance window on 2026-02-15, the cron job was disabled to prevent interference with infrastructure changes. A TODO to re-enable it was left in a Slack thread but never tracked as a task.

The certificate expired exactly on schedule. The auth service attempts a TLS handshake to the token signing service on startup, and the expired cert caused an immediate crash. Kubernetes restarted the pods, which crashed again, entering a CrashLoopBackOff.

- No monitoring existed for certificate expiry dates
- No graceful degradation path existed for auth service TLS failures
- The dependency between auth service and every other service was total — no service could function without real-time token validation

## IMPACT

- Complete platform outage for 1 hour 23 minutes
- Approximately 31,000 failed API requests
- Webhook delivery delayed by up to 2 hours for events during the window
- No data loss — all queued webhooks were delivered after recovery
- SLA breach for two enterprise customers (99.9% monthly uptime)

## REMEDIATION

### Immediate

1. Regenerated internal CA certificate
2. Re-enabled the certificate rotation cron job
3. Added calendar reminder as a backup for the cron

### Follow-up

- Add certificate expiry monitoring with 14-day and 3-day alerts
- Add a pre-flight check to the maintenance window runbook: list all disabled crons and verify re-enablement
- Investigate whether the auth service can cache validated tokens to survive brief signing service outages
- Add circuit breaker to auth service with a degraded mode that accepts recently-cached tokens

## LESSONS

- Disabling automation during maintenance is a deferred risk, not a deferred task. Every disabled cron should be tracked as an open item, not a Slack message.
- Certificate expiry is one of the most predictable failure modes in infrastructure. If you are not monitoring it, you will experience this incident.
- Total auth dependency with no degradation path means a single expired certificate takes down every service. The blast radius was disproportionate to the cause.
