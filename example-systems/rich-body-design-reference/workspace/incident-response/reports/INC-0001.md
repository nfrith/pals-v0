# Vector Cache Poisoning During Release 2026.03.18

A stale embedding cache let an outdated safety policy outrank the current release policy during tool selection. The issue surfaced only under retry pressure, so the first reports looked like intermittent flakiness instead of a deterministic routing fault.

This record is intentionally written as a rich operational document: short framing up front, then named sections that separate summary, chronology, technical evidence, and follow-up work.

## SUMMARY

Production agents selected a deprecated recovery tool after retries reused an invalidated cache shard. Traffic was diverted within 14 minutes and recovered without customer data loss.

### Detection

- The pager triggered when escalation latency crossed 90 seconds.
- On-call noticed that:
  - cold-cache requests behaved correctly
  - retries on hot shards selected the deprecated tool
  - the bad tool path only appeared after a second model invocation

### Recovery Lead Notes

> If the cached policy version and the prompt contract disagree, treat the cache as advisory, not authoritative.

## IMPACT

### Affected Flows

- Incident-recovery agents handling multi-step retries
- Escalation planners that depend on the latest safety policy manifest

### What Users Saw

- Slower escalation handoff for roughly 11 minutes
- One incorrect recovery recommendation that was blocked by a human approval gate

### What Did Not Break

- Fresh sessions with empty policy caches
- Read-only research assistants
- Stored customer data and audit logs

## TIMELINE

1. 09:02 UTC: alert fired for elevated escalation latency
2. 09:05 UTC: Ada confirmed the issue reproduced only on retry-heavy flows
3. 09:08 UTC: Malik compared tool traces and found policy-version divergence
   - retry path selected `recover_legacy_session`
   - cold path selected `recover_current_session`
4. 09:11 UTC: cache invalidation was forced for the affected shard family
5. 09:16 UTC: traffic normalized and manual sampling showed the correct tool path
6. 09:27 UTC: incident downgraded from active mitigation to monitoring

## TECHNICAL_ANALYSIS

### Failure Path

The router compared the live policy manifest to cached shard metadata, but the invalidation rule only triggered on major-version changes. The 2026.03.18 release changed recovery-tool precedence without changing the cache family's major key.

#### Cache Shard 07

- Cached manifest: `2026.03.04`
- Live manifest: `2026.03.18`
- Retry count at failure: `2`
- Divergence only appeared after the second embedding reuse

```yaml
policy_manifest: 2026.03.18
cached_manifest: 2026.03.04
cache_key: incident-router:v14:shard-07
selected_tool: recover_legacy_session
expected_tool: recover_current_session
retry_attempt: 2
```

### Why Alerts Lagged

> The metrics pipeline counted successful tool execution, not policy-version divergence, so the alert surfaced as latency before it surfaced as correctness drift.

### Contributing Conditions

- Release policy changed tool precedence without bumping the cache family
- Retry code reused the routing cache more aggressively than single-pass flows
- Approval gates prevented a worse outcome, but they also delayed recognition because the wrong tool never completed end-to-end

## RECOVERY

### Mitigation Steps

1. Freeze rollout of the new routing package
2. Force invalidation for the affected cache family
3. Replay the last 50 retry-heavy sessions against the corrected manifest
4. Re-enable rollout with a temporary policy-version parity check

```bash
routerctl cache invalidate --family incident-router:v14 --reason policy-version-drift
routerctl rollout pause --component recovery-router
routerctl replay --sample retry-heavy-50 --policy-manifest 2026.03.18
```

### Verification

- Replay batch chose the expected tool in all sampled sessions
- Live traffic stayed under the pre-incident latency threshold for 20 consecutive minutes
- No additional approval-gated mismatches were observed

### Stop Conditions

> If replay had shown mixed tool selection after invalidation, the next move would have been a full rollback instead of a guarded resume.

## FOLLOW_UP

### Immediate Actions

- Add manifest-version parity checks before cached routing results are reused
- Update the recovery runbook with a cache-poisoning branch
- Emit a dedicated metric for policy-version divergence
