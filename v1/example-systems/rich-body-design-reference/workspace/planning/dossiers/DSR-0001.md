# Delivery Plan For Recovery-Agent Escalation Guardrails

This dossier exists because the current recovery agent can usually find a valid next step, but it still has weak judgment around when to stop and ask for human approval.

The work is not only a prompt update. It spans evaluation design, incident-response policy, and runbook updates, so the plan needs explicit assumptions and alternatives instead of a loose TODO list.

## GOAL

Ship guardrails that make the recovery agent escalate sooner on ambiguous high-impact actions without making routine recoveries unusably slow.

### Success Boundary

- Fewer late escalations during incident replay
- No material increase in unnecessary approvals during low-risk recoveries

## ASSUMPTIONS

### Current Beliefs

- The agent already has enough context to detect many ambiguous actions.
- The primary failure is thresholding, not total lack of evidence.

### Fragile Assumption

> If adjudication costs explode, the team will disable the guardrail instead of tuning it, so operator review cost must stay visible in the rollout plan.

## OPTIONS

### Option A: Prompt-Only Threshold Shift

- Fastest to ship
- Lowest coordination cost
- Highest risk of prompt drift without measurable evidence

### Option B: Prompt Update Plus Eval Gate

- Slower initial ship
- Better confidence because behavior changes are replayed against a fixed eval set
- Still depends on operator policy being explicit

### Option C: Full Policy Engine Before Any Rollout

- Most principled long term
- Too slow for the current risk window

```text
Recommendation:
ship Option B first, then decide whether a separate policy engine is still justified.
```
