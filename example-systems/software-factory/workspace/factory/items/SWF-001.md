# SWF-001: Fix flaky migration replay in staging deploy

## DESCRIPTION

The staging deploy occasionally replays an already-applied migration and leaves the environment half-upgraded. This item captures the hotfix work needed to stabilize deploys before the next release window.

## CURRENT_STATE

- The failure reproduces only in staging.
- The replay path appears after a partial rollback and second deploy attempt.
- Operations currently recovers by manually clearing the stale migration lock.

## REQUIREMENTS

- The fix must stop duplicate replay without introducing a new migration ordering bug.
- The deploy path must stay reversible.
- The operator needs a clear recovery note if the stale lock already exists.

## DOD

- [ ] Staging deploy no longer replays an already-applied migration.
- [ ] Recovery path is documented.
- [ ] A regression check exists for the replay case.

## PLAN

### Proposed approach

- Trace the lock acquisition and release path in the migration runner.
- Add an idempotency guard before replay is attempted.
- Document the stale-lock recovery step in the deploy notes.

## PLAN_QUESTIONS

### 2026-04-05

- Should the hotfix patch only the runner, or should it also rewrite the stale lock left by partial rollback?
- Is the release operator comfortable taking a small schema-runner change in a hotfix branch, or should we ship a narrower guard first?
