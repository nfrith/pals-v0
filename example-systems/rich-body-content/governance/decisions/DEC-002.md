# DEC-002

## CONTEXT

Our background job processing is currently handled by in-process setTimeout calls and a single Redis-backed queue using Bull. This architecture has reached its limits.

- Jobs regularly fail silently when the API server restarts during deploys
- There is no visibility into job status, retry state, or failure reasons
- The Bull queue is a single point of failure — Redis downtime stops all background work
- We process approximately 50K jobs per day across 8 job types

## OPTIONS

### Dedicated Worker Service with BullMQ

Upgrade Bull to BullMQ and move job processing to a dedicated worker service separate from the API server.

Pros:

- Minimal migration effort — BullMQ is the successor to Bull with a compatible API
- Dedicated workers survive API server deploys
- Built-in dashboard (Bull Board) for job visibility
- Proven at our scale

Cons:

- Still depends on Redis as the single backing store
- BullMQ's concurrency model is per-worker, not per-job-type — noisy neighbor risk
- No built-in dead letter queue semantics

### PostgreSQL-backed Job Queue

Use our existing PostgreSQL database as the job queue via a library like Graphile Worker or a custom SKIP LOCKED implementation.

Pros:

- No additional infrastructure — jobs live in the database we already operate
- Transactional job creation — enqueue a job in the same transaction as the business logic
- Strong durability guarantees from PostgreSQL

Cons:

- Polling-based — higher latency than Redis pub/sub for time-sensitive jobs
- Database contention risk under high job volume
- Less ecosystem tooling than Redis-backed solutions

> Graphile Worker benchmarks: "Processes 10K+ jobs/second on a single PostgreSQL instance with SKIP LOCKED."

### Cloud-Native Queue Service

Adopt a managed queue service like AWS SQS or Google Cloud Tasks.

Pros:

- Zero operational overhead — fully managed
- Scales automatically with job volume
- Built-in dead letter queues and retry policies

Cons:

- Adds cloud provider coupling to the job processing layer
- SQS has at-least-once delivery — requires idempotent job handlers
- Latency is higher than in-process or Redis-backed solutions
- Local development requires mocking or emulator setup

## DECISION

Adopt the PostgreSQL-backed approach using Graphile Worker.

- Eliminates Redis as a dependency for background jobs, reducing operational surface
- Transactional job creation solves our biggest reliability pain: jobs that were enqueued but the business logic transaction rolled back
- The 50K jobs/day volume is well within PostgreSQL's capacity
- Graphile Worker provides a dashboard, retry logic, and cron scheduling out of the box
- We accept the polling latency tradeoff — none of our current job types are latency-sensitive below 1 second

## ACTION_ITEMS

1. Set up Graphile Worker in the existing PostgreSQL database with a dedicated schema
2. Migrate the three highest-volume job types first: email delivery, webhook dispatch, and report generation
3. Add a Grafana dashboard for job throughput, latency, and failure rates
4. Deprecate the Bull queue after all job types are migrated
5. Remove Redis dependency from the infrastructure once no services require it
