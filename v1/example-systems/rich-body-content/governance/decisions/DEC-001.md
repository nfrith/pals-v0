# DEC-001

## CONTEXT

We are preparing to ship breaking changes to the `/api/users` and `/api/billing` endpoints. We have never versioned the API before — all consumers use the same unversioned endpoints. We need to choose a versioning strategy before the breaking changes ship.

- There are currently 14 external API consumers and 3 internal services
- The breaking changes affect request body schemas and response envelope formats
- We want to support at least two versions simultaneously during migration periods

## OPTIONS

### URL Path Versioning

Embed the version in the URL: `/api/v1/users`, `/api/v2/users`.

Pros:

- Immediately visible in logs, documentation, and debugging
- Easy to route at the load balancer or API gateway level
- No ambiguity about which version a request targets

Cons:

- Pollutes the URL namespace
- Makes it tempting to version individual endpoints rather than the whole API surface
- Clients must update URLs, not just headers, when migrating

> GitHub, Stripe, and Twilio all use URL path versioning. It is the most common pattern for public APIs.

### Header Versioning

Version via a custom header: `Api-Version: 2`.

Pros:

- Clean URLs — the version is metadata, not part of the resource path
- Can default to the latest version for consumers that do not send the header

Cons:

- Invisible in browser testing and URL sharing
- Harder to route at the gateway level without header inspection
- Easy for clients to forget the header and accidentally hit the wrong version

> The default-to-latest behavior sounds convenient but is a trap. A client that forgets the header silently breaks when you ship a new version.

### Query Parameter Versioning

Version via query parameter: `/api/users?version=2`.

Pros:

- Visible in URLs and easy to test in a browser
- Does not require header manipulation

Cons:

- Mixes API metadata with resource parameters
- Caching proxies may not vary on query parameters correctly
- Looks unprofessional in documentation

| Option | Visibility in logs and docs | Client migration burden | Routing simplicity | Main drawback |
| --- | --- | --- | --- | --- |
| URL path versioning | High | Medium | High | Pollutes the URL namespace and forces explicit URL changes. |
| Header versioning | Low | Low | Low | Easy for clients to omit and hard to debug in a browser. |
| Query parameter versioning | Medium | Low | Medium | Weak caching semantics and poor documentation ergonomics. |

## DECISION

Adopt URL path versioning. The visibility benefit outweighs the URL pollution concern. Our API gateway already supports path-based routing, so no new infrastructure is needed.

- All endpoints will be mounted under `/api/v1/` immediately
- Breaking changes will ship under `/api/v2/` when ready
- Minimum support window for deprecated versions: 6 months
- Version retirement will be announced 90 days in advance via email and API response headers

## ACTION_ITEMS

1. Mount all existing endpoints under `/api/v1/` with backward-compatible redirects from the unversioned paths
2. Update API documentation to reflect versioned URLs
3. Notify external consumers of the versioning scheme and migration timeline
4. Add `Sunset` and `Deprecation` response headers to the framework middleware
5. Update internal services to pin their requests to `/api/v1/`

## NOTES

The header versioning option had support from the frontend team because it keeps URLs clean for their SPA routing. Their concern was acknowledged but the operational visibility argument won out.

> Frontend lead: "We can always abstract the version into our API client config. Clean URLs matter more for the browser address bar, which does not apply to API calls."
