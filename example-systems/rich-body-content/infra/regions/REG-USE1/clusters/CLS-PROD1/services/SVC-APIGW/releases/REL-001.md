# REL-001

## CHANGES

Adds rate limiting middleware and upgrades the Prisma client to v6.

- Added per-IP rate limiting at 100 requests/minute using a sliding window algorithm backed by Redis
- Upgraded Prisma from v5.22 to v6.0 — includes query batching improvements and updated type generation
- Fixed a bug where webhook retry headers were not parsed correctly for timestamps in non-UTC timezones

```typescript
// New rate limiter middleware
import { rateLimit } from "./middleware/rate-limit";

app.use(rateLimit({
  windowMs: 60_000,
  max: 100,
  keyGenerator: (req) => req.ip,
  store: new RedisStore({ client: redis }),
}));
```

## ROLLBACK_PLAN

> The Prisma v6 upgrade includes a migration that adds an index. Rolling back the application code does not roll back the index — this is safe, the old Prisma client ignores unknown indexes.

1. Revert the deployment to the previous image tag:

```bash
kubectl set image deployment/api-gateway \
  api-gateway=registry.example.com/api-gateway:2.13.1 \
  -n production
```

2. Verify rollback completed:

```bash
kubectl rollout status deployment/api-gateway -n production
```

3. Confirm rate limiting is no longer active by checking response headers — `X-RateLimit-Remaining` should be absent

## VERIFICATION

1. Check that pods are running the correct version:

```bash
kubectl get pods -l app=api-gateway -o jsonpath='{.items[*].spec.containers[0].image}'
# Expected: registry.example.com/api-gateway:2.14.0
```

2. Verify rate limit headers are present:

```bash
curl -I https://api.example.com/v1/health | grep X-RateLimit
# Expected: X-RateLimit-Limit: 100, X-RateLimit-Remaining: 99
```

3. Confirm Prisma query batching is active in logs:

```bash
kubectl logs -l app=api-gateway --since=5m | grep "prisma:query" | head -5
```
