# INC-003

## SUMMARY

Replica search indexes began returning stale result sets for a subset of tenants after a backfill job replayed old snapshots out of order. The issue is contained to search freshness so far, but the root cause and exact blast radius are still under investigation.

## TIMELINE

### Detection

- 08:14 - Support escalated multiple reports of missing newly-created documents in search
- 08:19 - On-call compared primary and replica result sets and confirmed replication drift

> Alert correlation is weak so far - freshness drift surfaced through customer reports before any automated signal fired.

### Investigation

- 08:27 - Replayed indexing metrics and found delayed replica catch-up on one shard family
- 08:34 - Compared snapshot metadata and found at least one replica applying an older backfill after a newer write stream
- 08:46 - Began tracing the ordering guarantees between the backfill worker and the live replication stream

### Current State

- Search writes remain enabled
- Affected tenants are being sampled manually
- The team has not yet committed to a mitigation strategy

## ROOT_CAUSE

The working theory is that the replica backfill worker can apply a stale snapshot after the live stream has already accepted newer writes for the same tenant shard. The exact ordering bug is not confirmed yet, and no single failing code path has been isolated.

- Possible interaction between snapshot replay and live stream acknowledgements
- No proven data loss, but read freshness is clearly degraded
- Investigation is focused on ordering guarantees, replay idempotency, and shard handoff timing

## IMPACT

- Search freshness is degraded for a subset of tenants
- Newly-created or newly-updated documents may be absent from replica-backed search results
- Direct document access is unaffected
- No evidence yet of corruption in the primary index

## REMEDIATION

### Immediate

1. Sample affected tenants and compare primary versus replica results
2. Pause any non-essential backfill work if it increases replica drift
3. Capture ordering evidence before changing the replication topology

### Follow-up

- Confirm whether snapshot replay can arrive after newer live writes for the same shard
- Decide whether to pause replica reads or force reindex on the affected shard family
- Add explicit freshness-drift alerting if the incident pattern is confirmed
