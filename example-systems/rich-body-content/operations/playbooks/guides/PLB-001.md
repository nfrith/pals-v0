# PLB-001 Stripe Payment Integration

## OVERVIEW

Stripe handles all payment processing for the platform. We use it for one-time charges, subscription billing, and refund management. This replaced the legacy PayPal integration in Q3 2025.

## AUTHENTICATION

All requests use bearer token auth with a restricted API key. We maintain separate keys for test and production environments.

> Never use the root secret key in application code. Always create restricted keys scoped to the operations you need.

Required environment variables:

```bash
STRIPE_SECRET_KEY=sk_live_...
STRIPE_WEBHOOK_SECRET=whsec_...
STRIPE_API_VERSION=2025-01-15
```

- The secret key must have `charges:write`, `refunds:write`, and `customers:read` permissions
- The webhook secret is generated when you register the endpoint in the Stripe dashboard
- Pin the API version to avoid breaking changes from Stripe's rolling updates

## ENDPOINTS

### Create Charge

POST to `/v1/charges` to create a one-time payment.

```bash
curl https://api.stripe.com/v1/charges \
  -u "$STRIPE_SECRET_KEY:" \
  -d amount=2000 \
  -d currency=usd \
  -d source=tok_visa \
  -d description="Order #1234"
```

Response on success:

```json
{
  "id": "ch_1abc123",
  "object": "charge",
  "amount": 2000,
  "currency": "usd",
  "status": "succeeded",
  "paid": true
}
```

### Retrieve Charge

GET `/v1/charges/{charge_id}` to check payment status.

```bash
curl https://api.stripe.com/v1/charges/ch_1abc123 \
  -u "$STRIPE_SECRET_KEY:"
```

### Issue Refund

POST to `/v1/refunds` to refund a charge. Partial refunds are supported.

```bash
curl https://api.stripe.com/v1/refunds \
  -u "$STRIPE_SECRET_KEY:" \
  -d charge=ch_1abc123 \
  -d amount=500
```

- Omit `amount` to refund the full charge
- Refunds take 5-10 business days to appear on the customer's statement
- A charge can be refunded multiple times up to the original amount

### Webhook Verification

Incoming webhooks must be verified using the signing secret.

```typescript
import Stripe from "stripe";

const stripe = new Stripe(process.env.STRIPE_SECRET_KEY);

function verifyWebhook(payload: string, signature: string): Stripe.Event {
  return stripe.webhooks.constructEvent(
    payload,
    signature,
    process.env.STRIPE_WEBHOOK_SECRET
  );
}
```

## ERROR_HANDLING

Stripe returns structured errors with a `type` and `code` field.

- `card_error` — the card was declined. Display the `message` to the user
- `rate_limit_error` — too many requests. Back off with exponential retry
- `invalid_request_error` — malformed request. Do not retry, fix the code
- `authentication_error` — bad API key. Check env vars
- `api_connection_error` — network failure. Retry with backoff

> Rate limits are 100 requests/second in live mode and 25/second in test mode. If you hit them, you have an architectural problem, not a rate limit problem.

Example error response:

```json
{
  "error": {
    "type": "card_error",
    "code": "card_declined",
    "message": "Your card was declined.",
    "param": "source"
  }
}
```
