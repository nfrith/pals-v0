# Credential Rotation After Token Exposure

Use this runbook when an agent-accessible token is exposed in logs, transcripts, or cached tool output. The goal is to rotate quickly without losing track of which downstream systems still trust the old credential.

Do not use this procedure for customer-managed keys or human SSO credentials. Those require a different approval path.

## PREREQUISITES

### Required Access

- Secret manager write access for the affected environment
- Deployment permission for the service that consumes the token
- Read access to the audit log stream

### Prechecks

- Confirm the exact token identifier, not just the service name.
- Confirm whether the exposed token has already been revoked by another operator.

```text
Minimum context bundle:
- token id
- owning service
- first known exposure location
- environments that currently trust the token
```

## EXECUTION

### Rotation Sequence

1. Mint a replacement token with the same scope.
2. Store the replacement token in the secret manager.
3. Roll the consuming service.
4. Revoke the exposed token only after the new token is live.

### Dry-Run Example

```bash
secctl token issue --service recovery-router --scope prod-ops
secctl secret put --path services/recovery-router/token --from-stdin
deployctl rollout restart recovery-router --env prod
secctl token revoke --token tok_live_exposed_01
```

### Safety Notes

> Never revoke first unless the service has a verified fallback path. A premature revoke turns a containment task into an outage.

### If Multiple Services Share the Token

- Rotate once, then deploy consumers in dependency order:
  - highest-request-volume services first
  - approval-gated services second
  - background maintenance jobs last

## VERIFICATION

### Required Checks

1. Confirm the new token appears in the consuming service environment.
2. Confirm the service can complete one authenticated health check.
3. Confirm the old token no longer authenticates.

```bash
deployctl env print recovery-router --env prod | rg TOKEN
curl -H "Authorization: Bearer $NEW_TOKEN" https://ops.example.test/healthz
curl -H "Authorization: Bearer $OLD_TOKEN" https://ops.example.test/healthz
```

## ESCALATION

### Stop And Escalate If

- A dependent service cannot reload credentials without a full restart.
- The new token authenticates intermittently.
- Audit logs suggest the exposed token was used after the reported exposure window.

### Escalation Targets

- Page Ada Torres if production recovery flows are degraded.
- Pull in the platform owner if secret propagation exceeds 10 minutes.

## NOTES

### Review Notes

- This procedure assumes token scopes do not change during rotation.
- Add a branch for bulk tenant-token rotation if shared credentials are reintroduced.
