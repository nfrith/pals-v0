# INC-001

## SUMMARY

All API requests began failing with 500 errors at 14:32 UTC on 2026-02-14 due to exhausted database connection pools. The outage lasted 47 minutes and affected all authenticated endpoints. Root cause was a leaked connection in the batch export job introduced two days earlier.

## TIMELINE

### Detection

- 14:32 — PagerDuty alert fires: API error rate exceeds 30% threshold
- 14:34 — On-call engineer acknowledges and begins investigation
- 14:36 — Confirmed all API pods returning 500s on any database-dependent route

> Alert payload: "api-production error_rate 47.3% > 30.0% for 2 minutes"

### Investigation

- 14:38 — Checked pod health: all pods running, no OOM kills
- 14:40 — Checked database: CPU at 12%, connections at max (200/200)
- 14:42 — Ran connection audit query, found 183 idle connections from the batch-export service
- 14:45 — Correlated with the batch export deploy on 2026-02-12

> Connection audit: "SELECT count(*), application_name FROM pg_stat_activity GROUP BY application_name" showed batch-export holding 183 of 200 connections

### Mitigation

- 14:48 — Killed idle batch-export connections manually via pg_terminate_backend
- 14:52 — API recovery confirmed, error rate dropped to baseline
- 14:55 — Disabled the batch export cron job to prevent recurrence
- 15:19 — Wrote and deployed hotfix for the connection leak

## ROOT_CAUSE

The batch export job opened a new database connection per chunk but only closed it on the success path. When a chunk had zero records (common for inactive tenants), the early-return skipped the connection cleanup.

```typescript
// BEFORE (leaking)
async function exportChunk(tenantId: string) {
  const conn = await pool.acquire();
  const rows = await conn.query("SELECT ...");
  if (rows.length === 0) return; // connection never released
  await writeToS3(rows);
  conn.release();
}
```

Over 48 hours, the leaked connections accumulated until the pool was fully exhausted. The connection pool had no idle timeout configured, so leaked connections persisted indefinitely.

- The bug was introduced in commit `a3f8c21` on 2026-02-12
- Code review missed the early return because the reviewer focused on the S3 write logic
- No integration test covered the zero-records path

## IMPACT

- All authenticated API endpoints returned 500 errors for 47 minutes
- Approximately 12,400 failed requests during the window
- No data loss or corruption — the database itself was healthy
- Three customers contacted support during the outage
- Internal dashboards were also affected since they use the same API

| Surface | Effect | Duration | Notes |
| --- | --- | --- | --- |
| Authenticated API | 500 errors on database-backed routes | 47 minutes | Every customer-facing authenticated endpoint depended on the exhausted pool. |
| Internal dashboards | Unavailable | 47 minutes | Dashboards shared the same API and failed for the same reason. |
| Customer support | 3 inbound tickets | During outage window | Reports aligned with the period before the hotfix deploy. |

## REMEDIATION

### Immediate

1. Killed leaked connections and restored service
2. Disabled the batch export cron job
3. Deployed hotfix wrapping connection usage in try/finally

```typescript
// AFTER (fixed)
async function exportChunk(tenantId: string) {
  const conn = await pool.acquire();
  try {
    const rows = await conn.query("SELECT ...");
    if (rows.length === 0) return;
    await writeToS3(rows);
  } finally {
    conn.release();
  }
}
```

### Follow-up

- Add connection pool idle timeout of 30 seconds
- Add monitoring alert for connection pool utilization above 80%
- Add integration test for the zero-records export path
- Audit all other pool.acquire() call sites for similar patterns

## LESSONS

- Connection pools without idle timeouts are time bombs. Leaked connections accumulate silently until the pool is exhausted.
- Early returns are a code smell in resource-acquisition contexts. Prefer try/finally or a wrapper that guarantees cleanup.
- The existing alert caught the symptom (error rate) but not the cause (connection count). We need resource-level alerts, not just outcome-level alerts.
